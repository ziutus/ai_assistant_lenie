# Security Incident: Vault Token in settings.local.json

**Date:** 2026-03-07
**Severity:** Medium
**Status:** Resolved
**Detection:** GitHub Push Protection

## What Happened

During a routine push of the `feat/pg18-multi-model-embeddings` branch, GitHub Push Protection blocked the push because `.claude/settings.local.json` contained a **HashiCorp Vault Root Service Token** (`hvs.REDACTED`) in plaintext.

The token was embedded in a Claude Code allowlist pattern:
```
"Bash(VAULT_ADDR=http://192.168.200.7:8210 VAULT_TOKEN=hvs.xxx curl:*)"
```

This pattern was auto-generated by Claude Code when the user approved a Vault-related curl command during a previous session. The approval pattern captured the full command including the real token value.

## Why Pre-Commit Hooks Missed It

| Tool | Result | Why |
|------|--------|-----|
| **gitleaks** (pre-commit) | Not detected | Default rules did not match Vault tokens embedded inside JSON string values within allowlist patterns |
| **TruffleHog** (pre-commit) | Not detected | Pre-commit mode scans staged diff only; token format inside a complex JSON structure was not matched |
| **GitHub Push Protection** | **Detected** | Server-side scanning with more aggressive pattern matching for known token formats (`hvs.` prefix) |

## Resolution Steps

1. **Removed token** from `.claude/settings.local.json` — replaced with wildcard pattern `VAULT_TOKEN=*`
2. **Squashed commits** — combined the fix with the original commit so the token never appears in pushed history
3. **Added `.claude/settings.local.json` to `.gitignore`** — this file may contain secrets in auto-generated allowlist patterns and should not be tracked
4. **Removed file from git tracking** — `git rm --cached .claude/settings.local.json`
5. **Added custom gitleaks rules** for HashiCorp Vault token prefixes in `.gitleaks.toml`:
   - `hvs.` — Service tokens
   - `hvr.` — Root tokens
   - `hvb.` — Batch tokens
6. **Created `scripts/scan-history-secrets.sh`** — script to scan unpushed commits with TruffleHog before pushing

## Token Rotation

The exposed token (`hvs.REDACTED`) is for a **local NAS Vault instance** (192.168.200.7:8210) not accessible from the internet. However, as best practice:

- [ ] Rotate the Vault root token on NAS
- [ ] Verify no other files contain this token

## Preventive Measures

### Already Implemented
- `.claude/settings.local.json` added to `.gitignore`
- Custom gitleaks rules for Vault token patterns (`hvs.`, `hvr.`, `hvb.`)
- `scripts/scan-history-secrets.sh` for pre-push history scanning

### Recommended
- Add `scan-history-secrets.sh` as a **pre-push hook** to catch secrets in all unpushed commits automatically
- Periodically run `trufflehog git file://. --no-update` for full repository history scan
- Review Claude Code allowlist patterns periodically for accidentally captured secrets

## Lessons Learned

1. **Claude Code allowlist patterns can capture real secrets** — when a user approves a command containing a token, the full command (including the secret) is stored in `settings.local.json`
2. **Pre-commit hooks are not sufficient alone** — they scan only the staged diff and may miss secrets in unusual formats or locations
3. **Defense in depth works** — GitHub Push Protection caught what local tools missed
4. **Always squash before pushing** when removing a secret — a separate "fix" commit still exposes the secret in the previous commit's history
