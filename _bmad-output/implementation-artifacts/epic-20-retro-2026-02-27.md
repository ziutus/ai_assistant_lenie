## What Went Well

1. **AI workspace security** — Sensitive data (API keys, database passwords, tokens) removed from the project directory accessible to AI agents. With `SECRETS_BACKEND=vault` or `aws`, `.env` contains only 7 bootstrap variables — no secrets exposed to AI tooling.

2. **Rapid execution** — 6 stories in ~33 hours. Well-planned dependency sequence: 20-1 (interface) → 20-2/20-3 (parallel backends) → 20-4 (integration) → 20-5/20-6 (docs & tooling).

3. **Elegant backward compatibility** — Non-env backends inject values into `os.environ`, so library modules using `os.getenv()` continue to work without modification. Enables incremental migration.

4. **Emergent story 20-6** — During Stories 20-3/20-4, the team discovered a gap: no tools to compare state between backends, no SSOT for variable classification. Instead of deferring, Story 20-6 was added to the epic and delivered with 36 unit tests and a comprehensive CLI toolset.

5. **66 new unit tests** — Solid safety net for the config_loader module and env_to_vault.py CLI tool. Significantly above average test coverage for an epic.

6. **B-64 completed** — Pre-commit secret detection verified end-to-end (from Epic 19 retro action item). Gitleaks detects 6/10 secret types, TruffleHog verifies real secrets online. Custom `.gitleaks.toml` rules added for GitHub PAT.

7. **SSOT for configuration** — `scripts/vars-classification.yaml` defines all ~50 config variables with classification (secret/config), backend definitions, and environment mappings. Foundation for ongoing variable lifecycle management.

8. **Comprehensive documentation** — `docs/secrets-management.md` created covering architecture, Vault setup, AWS SSM setup, variable classification, adding new variables, and troubleshooting.

## What Could Be Better

1. **Story files missing again** — 5 out of 6 stories have no file in implementation-artifacts. Only 20-5 has a full Dev Agent Record. This is the third consecutive epic with this pattern (after Epic 18 and 19).

2. **Story file policy still not formalized** — Action item from Epic 19 retro (Bob's responsibility) was not completed. No written policy defining when story files are required vs optional.

3. **Standalone scripts not migrated** — `imports/unknown_news_import.py`, `imports/dynamodb_sync.py`, and batch scripts still use `load_dotenv()` directly instead of config_loader. Discovered during retrospective — would have caused failure during planned Vault testing.

4. **VPC Lambda constraint** — `sqs-to-rds` Lambda cannot reach SSM without NAT Gateway (cost constraint). Keeps CloudFormation environment variables as a workaround. Same constraint blocks B-23 (IPv6 dual-stack).

5. **Story 20-6 lacks visible code review** — In a security-sensitive epic, all stories should have documented review. 4/6 is good but not 100%.

## Key Learnings

- **Security priority justifies speed over documentation** — When sensitive data is exposed to AI agents, removing it fast is the right call. Documentation catches up later.
- **Backward compatibility via os.environ injection** — Elegant pattern that enables incremental migration without breaking existing code. Library modules continue using `os.getenv()` while entry points use `cfg.require()`.
- **Real-world testing is critical** — Unit tests (66) verify individual components, but importing real data through the full pipeline is the true integration test.
- **Standalone scripts are a separate migration layer** — server.py and Lambda handlers were migrated, but standalone scripts (imports, batch processing) were out of scope. These need explicit attention.
- **Emergent scope discovery is healthy** — Story 20-6 arose from a real need discovered during implementation. Adding it to the epic rather than deferring delivered high-value tooling.

## Significant Discoveries

**Standalone scripts gap:** Import scripts (`unknown_news_import.py`, `dynamodb_sync.py`) and batch processing scripts still use `load_dotenv()` directly. They will fail when `.env` contains only bootstrap variables (Vault/AWS mode). This was discovered during retrospective discussion and added to critical path for immediate fix.

**No epic-level changes required** — The discovery affects operational scripts, not architecture or future epic planning. Fix is straightforward (replace `load_dotenv()` with `load_config()`).
