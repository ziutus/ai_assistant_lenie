## What Could Be Better

1. **Assumptions about existing state** — 17-2 assumed `Response` was imported in Flask (it wasn't). 17-3 assumed Lambda was in eu-central-1 (it was us-east-1). Epic docs stated "8 endpoints" but actual count was 9 URIs. Pattern: trust documentation over verification.
2. **Documentation drift persists** — 17-3 code review found HIGH: `infrastructure-metrics.md` had stale counts (26→27, 33→34). Same problem as Epic 16, despite having `verify-documentation-metrics.sh`.
3. **verify-documentation-metrics.sh degraded by noise** — 4 pre-existing FAILs on Windows/MSYS mask new real failures. When a test suite always reports failures, its signal value is lost.
4. **Edge cases missed in first implementation pass** — 17-1: `--yes` didn't skip delete confirmation. 17-2: no regression test for `/healthz`. Both caught by code review, but should have been caught earlier.
5. **Missing story file for 17-4** — Only story in the epic without implementation documentation. Lessons from that complex story are partially lost.

## Key Learnings

- **"Verify before assuming"** — Always check actual AWS state (region, resources, imports) before implementing. Don't trust documentation blindly.
- **Test suite with known failures loses signal** — Either fix all failures or clearly mark/skip known issues. A perpetually-failing test is worse than no test.
- **Code review is the reliable last line of defense** — 14+ findings prove its value. But goal should be catching issues earlier (developer self-review of edge cases).
- **Retrospective follow-through works** — 100% completion of Epic 16 action items demonstrates the value of tracking commitments across epics.
- **Pattern reuse accelerates delivery** — 17-1 reused zip_to_s3.sh safety pattern, 17-3 reused parameterized naming from Story 15-1. Established patterns reduce implementation risk.
