## What Went Well

1. **Speed of execution** — zero blockers, both stories completed smoothly. Strong momentum from Epics 26-27 foundation work.
2. **Dual-mode constructor pattern** — established in Story 27.2, proved repeatable and predictable. Adding ORM branches was mechanical.
3. **SQL injection fixes** — B-86 partially resolved as a side effect of ORM migration (parameterized queries replace f-string interpolation).
4. **Consistent test pattern** — session mocking approach is repeatable and readable across all ORM stories.
5. **No new dependencies** — no `.venv_wsl` sync needed, no build changes.

## What Could Be Improved

1. **Mixed uncommitted changes in working tree** — Story 28.1 review flagged as CRITICAL that changes from Stories 27.x were mixed with 28.1 in the same uncommitted tree. Made review scope unclear.
2. **Pre-existing test ordering issue** — 8 tests in `test_get_list_query.py` fail only in full suite due to `sys.modules` psycopg2 mock isolation. Carried forward unresolved from previous stories.
3. **Enum handling at ORM boundaries** — both stories required attention to enum serialization (ORM returns Python enum members, legacy returns strings). Story 28.2 review caught missing `document_type` enum-to-string conversion.

## Key Insights

1. Well-designed foundation (Epics 26-27) drastically accelerates subsequent epics — investment in patterns pays off.
2. Dual-mode constructor is the right migration strategy — allows incremental migration without breaking existing consumers.
3. Commit changes per story before code review — prevents scope confusion.
4. Enum handling needs explicit checklist when writing ORM methods that return dicts.
