# Retrospective: Epic 34 — Global Document Identifier (B-102)

**Date:** 2026-04-08
**Scope:** Sprint 13 (Epic 34) — Rename s3_uuid to uuid, backfill UUIDs, enforce NOT NULL + UNIQUE
**Stories completed:** 2/2 (100%)
**Facilitator:** Bob (Scrum Master, AI)

## Executive Summary

Epic 34 delivered a clean database column rename (`s3_uuid` -> `uuid`) with auto-generation, NOT NULL, and UNIQUE constraints (Story 34-1), followed by a mechanical code rename across all backend scripts and documentation (Story 34-2). Completed in one day with zero blockers, zero production incidents, and full NAS verification. Implements [ADR-015](docs/adr/adr-015-uuid-as-global-document-identifier.md).

## What Went Well

### 1. Blazing Fast Implementation
Entire epic completed in one day — two stories, zero blockers. Precise story files with exact file paths, line numbers, and occurrence counts eliminated guesswork and enabled rapid execution.

### 2. Smart Scope Split
Story 34-1 handled the risky part (Alembic migration, ORM model, SQL init, query layer) while 34-2 was a safe mechanical rename. This separation minimized risk at each step.

### 3. Proactive Scope Extension in 34-1
`stalker_web_documents_db_postgresql.py` was originally 34-2 scope, but 34-1 pulled it in because the query layer directly depends on the ORM attribute name. This prevented a broken query layer between stories.

### 4. Code Review Caught Silent Data Loss
Code review for 34-1 discovered that `document_service.py` was writing to `doc.s3_uuid` — a non-mapped attribute after the ORM rename. Python doesn't raise an error for this, so UUID data would have been silently lost. Fixed immediately.

### 5. DynamoDB Backward Compatibility
`dynamodb_sync.py` properly handles both old (`s3_uuid`) and future (`uuid`) DynamoDB field names with `item.get("uuid") or item.get("s3_uuid")`. Same pattern applied in `web_documents_do_the_needful_new.py` for input JSON.

### 6. Comprehensive Testing
587 tests passed after 34-1 (including ORM tests), 70 unit tests passed after 34-2 with zero regressions. NAS verification confirmed migration applied correctly.

## What Could Be Improved

### 1. Silent Attribute Assignment in Python/SQLAlchemy
Python doesn't raise errors when assigning to non-existent attributes on SQLAlchemy model instances. This caused the `document_service.py` bug caught in code review. On MVP, code review is sufficient safety net — no additional tooling needed now.

### 2. File List Automation (Carried Forward)
Fourth consecutive retro mentioning this. File Lists in Epic 34 were accurate and detailed, but the process is still manual. Low priority during MVP phase.

### 3. Lambda Code Still Uses s3_uuid
AWS Lambda files (`sqs-into-rds`, `sqs-weblink-put-into`) still reference `s3_uuid`. Intentionally deferred — AWS deployment is on hold for single-user NAS setup. Tracked in [`docs/aws-sync-backlog.md`](docs/aws-sync-backlog.md).
