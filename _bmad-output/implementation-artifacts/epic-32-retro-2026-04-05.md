# Retrospective: Epic 32 — Service Layer Extraction

**Date:** 2026-04-05
**Scope:** Sprint 11 (Epic 32) — Extract DocumentService, SearchService, migrate import scripts
**Stories completed:** 3/3 (100%)
**Facilitator:** Bob (Scrum Master, AI)

## Executive Summary

Epic 32 extracted business logic from Flask route handlers and import scripts into two reusable service classes: `DocumentService` (8 methods) and `SearchService` (2 methods). All 3 stories completed in ~6 days with zero blockers, zero production incidents, and 37 new unit tests (550 → 587). The established pattern from Story 32-1 accelerated subsequent stories. Code review found 13 issues (2H/11M/4L), all H+M fixed. The service layer provides a clean, Flask-independent interface ready for MCP server (B-57) and future consumers.

## What Went Well

### 1. Pattern Reuse Across Stories
The service extraction pattern established in 32-1 (DocumentService) became a direct template for 32-2 (SearchService) and 32-3 (import migration). Story 32-2 was implemented the next day with zero debug issues — rare velocity for a second story in an epic.

### 2. Clean Architectural Evolution
`server.py` route handlers reduced to 10-20 lines each (HTTP parsing + response formatting only). Removed unused imports (`os`, `uuid`, `WebDocument`, `text_transcript`, `text_functions`, `website_download_context`). The codebase is evolving from a monolithic Flask app to a layered architecture with clear separation of concerns.

### 3. Code Consolidation
`_store_file()` in DocumentService consolidated ~64 lines of duplicated S3/local storage code into a single private method. `import_document()` unified 3 different document creation paths (DynamoDB sync, RSS feeds, SQS messages) into one service method.

### 4. File List Tracking Fixed
All 3 stories have detailed, accurate File List sections in their story records. This was flagged in 3 consecutive retrospectives (Epic 26-30, Epic 31, now resolved).

### 5. Test Coverage Growth
37 new unit tests across the epic. Test suite grew from 550 to 587. Zero regressions. Each story maintained ruff-clean code.

## What Could Be Improved

### 1. Error Handling Gaps (Recurring)
Missing `session.rollback()` in exception handlers (32-1: H1, M3) and missing `try/except` entirely (32-2: M2) — caught by code review, not during development. This appeared in 2 of 3 stories.
**Decision:** Accepted at current scale (single-user). Will become critical for Phase 9 (multi-user). Documented as tech debt.

### 2. Service Instantiation in Loops
32-3 M1: `DocumentService` and `SearchService` were created per-item inside loops instead of once before the loop. Not a performance concern at current scale but indicates a pattern to watch.
**Decision:** Fixed in code review. Not a priority action item.

### 3. Test Count Reporting
32-3 M3 had inaccurate test count claim (also flagged in Epic 31 retro for Story 31-2). Minor trust issue in story records.
**Decision:** Low priority, noted for awareness.
